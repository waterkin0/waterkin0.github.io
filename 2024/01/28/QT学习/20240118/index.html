7.waitForConnected阻塞问题

我这里客户端想用waitForConnected来等待三秒来连接，但是这个阻塞函数会导致前面的setText等和ui有关的操作也阻塞

原因是因为和ui有关的操作要调用QWidget 的 paintEvent() 函数，而这个函数的自动调用是在程序main的主循环中进行

也就是说一切和ui有关的操作都在该循环结束时才会进行，那这里在waitForConnected阻塞后ui操作也被阻塞自然也就能理解了

一开始想用多线程，将waitForConnected放入新线程，防止阻塞，但太复杂了

正如上面所说，最简单的就是手动更新当前的ui，即 repaint()即可

但qt的主题是异步操作，不建议用阻塞函数，后续可能想办法用信号槽之类的优化吧

参考文献：qt5 TCP客户端 使用waitForConnected函数后，前面设置控件的代码不执行

8.还是中文编码问题

我这里问题主要是服务端和客户端通信的时候，两边编码解码的问题

简单总结就是以 local8bit来传递，在两端接收之后再解码即可

服务端：QString message = QString::fromLocal8Bit(socket->readAll());

客户端：const QByteArray message = ui->textEdit->toPlainText().toLocal8Bit();

将上面的message进行传递即可

9.按键检测(监听)问题

我想在textedit内编辑完后enter就可以直接调用某个函数，为此我需要一个办法来监听键盘

首先能查到的是 keyPressEvent这个函数，他是QWidght内的一个虚函数，在键盘上有某个按键按下之后会自动调用这个函数

但是这个函数扯到一个概念叫焦点，也就是说要用户的焦点在这个模块上，那么该函数才能被调用

这个很好解决，把输入框设置为焦点就好了，这里就不多做介绍

但实操发现还是不行，发现是textedit会劫持键盘上的事件，致使你在输入时所按的enter键会被识别为\n而不是触发上面那个函数

那这个办法暂时行不通，于是又找到了另一个东西——事件

QT很好的给了一个event事件供我们使用，当程序运行中，总是会触发各种事件，那么我们可以重写虚函数来过滤和处理我们想要的事件，很好的是，按键检测也在其中

我这里给出一个例子：

所有事件在产生之后会首先发给这个函数，然后再进行后续处理

可以看见这是一个bool类型的函数，返回值代表处理成功与否，如果返回true代表处理完毕，那么这次产生的事件不会继续传递至原来的地方，false则正常进行后续处理

一个简单的例子是你可以设置为检测到KeyPress后就返回true，那么你在textEdit内的操作都会被这个函数处理，导致你无论怎么敲键盘TextEdit内都不会有内容

但问题又来了，我按enter键似乎检测不到，不会进行处理

经过一番排查发现，qt将键盘上的enter分为两个，一个叫return，位于字母键盘上，即我们常用的enter，另一个在数字键盘旁边，这个才被叫enter

那在判断的时候就需要将两个都写入了

e->type() == QEvent::KeyPress && ((QKeyEvent*)e)->key() == Qt::Key_Enter || ((QKeyEvent*)e)->key() == Qt::Key_Return

现在正常了

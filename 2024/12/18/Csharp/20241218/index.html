有关C#内存管理的探讨

水帝 发布于 2024-12-18

📖1k字 ⏱3分 👀次

最后更新于 2024-12-26

  • 📒 Csharp
  • 🏷️ 内存管理

与C++不同的是，C#这一由微软封装的很好的语言几乎不需要自己进行内存管理，垃圾回收器会自动处理所有的内存清理工作，但这并不意味着我们不需要去理解其底层内存管理原理，我认为了解其底层无论如何都是对未来大有好处的

尽管C#没有指针，但本文还是用了大量指针的概念，以方便理解

1. 堆栈

以32位机器举例，其能为每个进程分配4gb的虚拟内存，这个4GB的内存包含了程序的所有部分（包括可执行代码、代码加载的所有DLL、程序运行时使用的所有变量的内容），而64位的虚拟内存高达16tb

与C++一致(或者说与几乎所有编程语言类似)，其内存管理的核心仍然是堆栈，栈从高向低填充，堆反之,两者顶部都有一个指针

对于每个线程，都有一个独立的栈，而堆是所有线程共享的

与数据结构的栈一样，内存栈也是先进后出(FIFO)，所以会优先释放后声明的变量

在C#内数据被分为值类型和引用类型，在C++内叫作值和指针，C#内指针管理被其自动托管，尽管我们看不到指针，但他仍然存在于该语言中

值类型往往占用较小的空间，他们往往是C#预定义的数据类型，例如int(4字节),double(8字节)等，而引用类型一般会占用较大的空间，例如自定义的类或者list等

在C#内值类型不能为空(除非用？，例如int? a，这回让其默认为null)，而引用可以为空

值类型存在与栈中，而引用类型本身存在于栈中，其引用的真正实例被分配在托管堆上

当需要存数据在堆内时，系统会搜索整个堆以找到连续的能够容纳该数据对象的内存块，然后存入

而在进行垃圾回收的操作之后(删除堆内已经没有被引用的数据)，系统会重新对堆内数据进行排序并重新从堆顶排列

栈内元素在使用完后立刻释放，而堆内则交给了垃圾回收GC

2. 垃圾回收GC

垃圾回收的时机由系统决定，在C#内由system.GC管理，必要时可以用System.GC.Collect()强制调用垃圾回收

但注意，垃圾回收是降低进程的性能，因为其在运行时会暂停所有线程的运行

一般堆会分为两部分，GC垃圾回收堆和加载堆：

  1. 垃圾回收堆(GC)：采用的是世代垃圾回收，分为0/1/2代三个部分，当0代满了之后会触发垃圾回收并将剩余的数据放入1代内，以此类推，它唯一的目的就是提升应用程序的性能，因为回收部分的性能明显优于回收整个堆，每一代的内存大小由系统自动分配
  2. 大对象堆(Large Object Heap，或者说加载堆)：大于85000字节的大对象会分配到这个区域，这个区域的主要特点就是：不会轻易被回收；就是回收了也不会被压缩（因为对象太大，移动复制的成本太高了）

3. 值类型和引用类型

两者可以总结为以下几点：

  1. 值类型：在C#中，继承自System.ValueType的类型被称为值类型，bool byte char decimal double enum float int long sbyte short struct uint ulong ushort`
  2. 引用类型：继承自System.Object， class interface delegate object string
